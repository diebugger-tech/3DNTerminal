# 3DNTerminal Architecture

## Core Philosophy
3DNTerminal is built on three pillars: **Atmosphere**, **Accessibility**, and **Security**.

## 1. Skill-Based Plugin System
The UI is driven by a modular `TerminalSkill` trait. 
- **Decoupling**: The main rendering loop (`two_d.rs`) only knows how to draw basic shapes and text. All high-level logic (Menus, Overlays, Effects) is encapsulated in Skills.
- **State Management**: Skills modify a central `Config` struct, which is then passed back to the renderer.

## 2. Security & AI Firewall (Human-in-the-Loop)
As 3DNTerminal evolves towards AI integration (Antigravity/Ollama), security is paramount.
- **AI-Firewall**: A dedicated `SecuritySkill` monitors all automated actions.
- **Human-in-the-Loop**: No command generated by an AI agent or automated script can be executed on the PTY without explicit manual approval via a secure overlay.
- **Privilege Separation**: The GUI frontend runs with standard user permissions. System-level actions are delegated to native OS mechanisms (Polkit).

## 3. Accessibility Engine (Barrierefrei)
- **Signal Damping**: A low-pass filter on the cursor input to assist users with tremors.
- **Color Mapping**: Real-time LUT (Look-Up Table) transformations for color blindness.
- **Motion Reduction**: A central flag to dampen or disable autonomous animations.

## 4. Rendering Pipeline
- **2D Canvas**: Powered by `iced::widget::canvas`.
- **Performance**: Double-buffered grid rendering to minimize redraws.
- **Hologram Effects**: Multi-layered stroke rendering for neon glow and glassmorphism.

## 5. Backend (Future)
- **PTY**: Integrated via `alacritty_terminal` for high-performance parsing.
- **AI**: Local integration via Ollama.
